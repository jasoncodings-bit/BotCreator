# Botivo desktop (Electron wrapper)

A thin Electron shell that runs the existing static Botivo web app (in the
repo root) as a native Windows desktop `.exe`. **This folder is the only
Electron-specific code** — the core app in the root still has no build step
and still runs fine served over HTTP or opened as `file://` in a browser.

It wraps **only the UI**. It does **not** start the local model server
(`llama-server`) — start that separately as before (see `../launch.bat`).
The app talks to `http://127.0.0.1:8081` exactly as it does in a browser.

## Run it in dev (no build)

```
cd desktop
npm install      # one-time: pulls Electron (this DOES need npm, unlike the web app)
npm start        # opens the app in a native window, loading ../index.html live
```

Because dev mode loads `../index.html` directly, any edit to the web app's
HTML/CSS/JS shows up on reload — no rebuild needed. DevTools stays enabled in
dev mode (it is blocked in the packaged build, see `main.js`).

## Rebuilding the .exe (the process that actually works)

We build with **`@electron/packager`**, not `electron-builder`. electron-builder's
Windows installer target fails on this machine with a symlink-permission error
(`Cannot create symbolic link … winCodeSign`) unless Windows Developer Mode is
on. `@electron/packager` just copies the Electron runtime + our files into a
folder — no signing, no symlinks — so it's the reliable path.

**Every time you change the web app (root `index.html`/`css`/`js`) you must
re-stage and repackage** — the exe bundles a *copy* of the app, so source
edits don't reach an already-built exe until you rebuild. Steps:

```
cd desktop

# 1. kill any running copy (it locks files in release/)
powershell "Get-Process Botivo,electron -ErrorAction SilentlyContinue | Stop-Process -Force"

# 2. re-stage the CURRENT web app into build-stage/webapp
rm -rf build-stage release
mkdir -p build-stage/webapp
cp -r ../index.html ../css ../js build-stage/webapp/

# 3. package (ELECTRON_RUN_AS_NODE must be unset, or electron runs as plain node)
env -u ELECTRON_RUN_AS_NODE ./node_modules/.bin/electron-packager . Botivo \
  --platform=win32 --arch=x64 --out=release --overwrite \
  --app-version=1.1.0 --icon=icon.ico \
  --extra-resource=build-stage/webapp \
  --ignore="(release|build-stage|electron-builder\.yml)"
```

Output: **`desktop/release/Botivo-win32-x64/`** — run `Botivo.exe` inside it.
The whole web app is copied to `resources/webapp/` in that folder; `main.js`
loads it from there when packaged (and from `../index.html` in dev).

### Make the shareable zip

The exe needs the whole folder next to it, so share the zip, not the lone exe:

```
powershell "Compress-Archive -Path 'release/Botivo-win32-x64/*' -DestinationPath 'release/Botivo-Windows.zip' -Force"
```

Friends unzip it anywhere and double-click `Botivo.exe`. (Unsigned, so Windows
SmartScreen shows a "protected your PC" prompt → *More info → Run anyway*.)

### Verify the rebuild picked up your changes

Quick sanity check that the exe's bundled copy is fresh, not stale:

```
grep -c "bot-export-modal" release/Botivo-win32-x64/resources/webapp/index.html
```

Then launch `Botivo.exe` and click through whatever you changed.

## The icon

`icon.ico` (multi-size) + `icon.png` are the app's own hand-drawn **robot**
(the `bot` icon from `js/icons.js`) — accent-orange strokes on the dark brand
badge. They're generated by rendering that SVG through Electron's Chromium and
screenshotting it at each size. If you change the artwork, regenerate the
`.ico`/`.png`, then repackage (packager only picks up the icon at build time).

## Notes

- `--app-version` is the exe's version — bump it when you ship an update.
- `node_modules/`, `release/`, and `build-stage/` are gitignored — build artifacts.
- The packaged build disables DevTools and the native menu (`main.js`), so end
  users can't pop open DevTools with Ctrl+Shift+I / F12.
